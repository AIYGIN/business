# 提案書ドラフト: 高配当分析画面 PoC

## 0. エレベーターピッチ

### 30秒版
本提案は、個人利用/PoCとして、日本株の高配当銘柄について「配当が持続しやすいか」を短時間で確認できる分析画面を作るものです。既存のスクリーニングサービスでは配当利回りやPER/PBRなどは確認できますが、FCF、減配履歴、増配率、配当性向、財務指標を同じ基準で100点化し、理由まで一画面で把握する用途には不足があります。v1では対象銘柄を20社固定とし、J-Quants APIを第一候補にしながら、初期はmock/手動データでも開始できる構成にします。画面リクエストごとに外部APIを呼ばず、「Data Fetch → Raw Data → Normalized Data → Score Calculator → BFF → FE」の流れで、保存済み分析データをBFFから返却します。AIサマリーはTODO扱いとし、売買推奨や将来予測を行わせない方針だけ先に固定します。PoCとして、データ取得スパイク、正規化、スコア計算、BFF API、FE表示の実装着手を承認してください。

### 1行版
高配当株の配当持続性を、FCF重視の100点スコアと理由説明で見える化するPoC提案。

### 決裁者にお願いしたいこと
- 高配当分析画面を個人利用/PoCとして実装開始すること。
- v1は20銘柄固定、J-Quants API第一候補、初期mock可、非リアルタイム更新で進めること。
- AIサマリーはv1ではTODO扱いとし、実装時には売買推奨・将来予測を禁止する方針を承認すること。

### なぜ今か
- 高配当銘柄は利回りだけでは持続性を判断しにくく、FCFや減配履歴を含めた確認が必要である。
- 画面モックとスコア設計が既にあり、PoC範囲なら実装対象を明確に切れる。
- J-Quants APIにより、日本株の公式系データ取得を検証しやすい環境がある。

### やらない場合の不利益
- 利回りの高さだけを見て、減配リスクやFCF不足を見落とす可能性が残る。
- 既存スクリーニングとの差別化ポイントである「配当持続性の説明UI」を検証できない。
- データ取得・AI文言・投資助言誤認リスクを早期に洗い出せない。

## 3. SCQAによる論理補強

### S: 現状
高配当銘柄は、既存サービスでも配当利回り、PER/PBR、ROE、配当性向などで検索できる。ただし、配当の持続性をFCFや減配履歴まで含めて100点化し、理由を同じ画面で説明するUXはPoC対象として独自価値がある。

### C: 課題
利回りが高い銘柄ほど、業績悪化や一時的な株価下落により見かけ上高利回りになっている可能性がある。利回りだけを見ると、減配リスクやキャッシュ創出力の弱さを見落とす可能性がある。

### Q: 問い
どうすれば、個人利用/PoCの範囲で、高配当株の配当持続性を短時間で安全に確認できるか。

### A: 提案
20銘柄固定のPoCとして、J-Quants APIを第一候補にデータ取得を検証し、初期はmock/手動データでもRaw Data、Normalized Data、Score Calculator、BFF、FEの流れを実装する。AIサマリーはTODO扱いとし、将来実装時も分析結果の説明に限定し、売買推奨・将来予測・投資判断を禁止する。

## 4. スコア設計

### 配点
| 項目 | 配点 | 意図 |
|---|---:|---|
| FCF | 30 | 配当原資の持続性を最重視する |
| 減配履歴 | 20 | 過去の株主還元安定性を見る |
| 増配率 | 15 | 無理のない増配かを見る |
| 配当性向 | 15 | 利益に対する配当負担を見る |
| 利回り | 10 | 高すぎる利回りを警戒する |
| 財務指標 | 10 | PER/PBR/ROEを補助的に見る |

### 最終判定
| スコア | 判定 |
|---:|---|
| 80〜100 | 安全寄り |
| 60〜79 | 注意しつつ良好 |
| 40〜59 | 注意 |
| 0〜39 | 危険 |

### 注意点
- このスコアは投資判断ではなく、過去データに基づく分析補助である。
- FCF、配当性向、増配率、減配履歴の算出定義は実装前に固定する。
- 金融業（三菱UFJ FG、三井住友FG、みずほFG、東京海上HD、MS&AD）は一般企業と同一のFCF評価を行わず、FCFはN/A（対象外）として扱い、残りの指標で100点換算する。

## 7. データ取得方針

### 第一候補: J-Quants API
- 日本株の株価・財務・配当等を取得する第一候補。
- 個人利用/PoCでは検証しやすい。
- ただし、利用条件は私的利用が前提であり、第三者配信や継続反復的な分析結果提供は制限がある。

### 補完候補: EDINET/APIまたは手動CSV
- J-Quantsで取得しにくい項目、または定義確認が必要な項目はEDINETや手動CSVで補完する。
- EDINETは公式開示資料に近いが、XBRL解析・タグ差異・正規化の保守負荷が高い。

### v1推奨
- 画面表示のたびに外部APIを叩かない。
- Data Fetchで事前取得した結果、または初期mock/手動データを保存し、BFFが保存済みデータだけを返す。
- 画面には最終更新日時と「リアルタイムではない」旨を表示する。

### v1実装フロー

```text
Data Fetch
    ↓
Raw Data
    ↓
Normalized Data
    ↓
Score Calculator
    ↓
BFF
    ↓
FE
```

- Step1: J-Quantsを第一候補としてRaw Dataを取得する。一旦はmock/手動データでもよい。
- Step2: Raw DataからFCF、配当性向、増配率、減配履歴、PER、PBR、ROEなどのNormalized Dataを生成する。
- Step3: Normalized Dataをもとにルールベースで `totalScore`、`judgement`、`scoreBreakdown`、`analysisSummary` を生成する。ただし `analysisSummary` はv1ではTODO可。
- Step4: BFFは保存済みデータのみを返却し、画面リクエスト中に外部APIへ同期アクセスしない。
