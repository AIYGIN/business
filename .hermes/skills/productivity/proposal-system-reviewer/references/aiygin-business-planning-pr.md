# AIYGIN/business planning PR 実例メモ

## 目的

AIYGIN/business に提案書を `planning/` 配下へ追加し、PR化したうえでシステム観点レビューコメントを残すときの実例パターン。

## 成功した流れ

1. 提案書ドラフトを作成する。
2. サブエージェントに「システム観点レビュー」章の初稿を作らせる。
3. 親エージェントが章を読み返し、投資助言・法務判断・未確認事項の断定がないか確認する。
4. 提案書へ統合する。
5. `AIYGIN/business` で `planning/<topic>` ブランチを切る。
6. `planning/<YYYYMMDD>-<short-topic>.md` として追加する。
7. `git add` は対象提案書だけに限定する。
8. commit / push / PR作成を行う。
9. PRにシステム観点レビューコメントを投稿する。
10. PR URL、コメントURL、commit SHA、追加ファイルを確認して報告する。

## 注意点

- リポジトリに既存の未コミット変更がある場合、今回のPRに含めるかを必ず切り分ける。
- 提案書PRでは、無関係な `.hermes/skills/` 変更を巻き込まない。
- PRコメントは、本文の要約ではなく「レビュー観点として決めるべきこと」を中心にする。
- システム観点レビューでは、mock / 無料API / cronスクレイピングを比較し、v1初期の推奨方針を明示する。

## PRコメントに最低限含めるもの

- 推奨BE方針
- 未確認の最重要リスク
- セキュリティ上の注意
- 障害時の対策方針
- キャッシュ導入判断
- 人間レビューで決めるべき事項
