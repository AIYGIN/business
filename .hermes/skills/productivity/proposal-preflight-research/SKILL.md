# 提案前 事前調査サブエージェント運用

## 目的

提案書を作り始める前に、別サブエージェントで以下を調査し、ユーザーが「この提案を進める価値があるか」を判断できる材料を作る。

- コスト
- リスク
- 実現可能性
- 既存サービス・代用手段

このスキルは、提案書本文を書くスキルではない。提案書作成前の意思決定材料を作るために使う。

## 使うタイミング

以下の依頼で使う。

- 「提案書を作る前に調査して」
- 「コスト、リスク、実現可能性を見て」
- 「似たサービスや代替手段がないか調べて」
- `proposal-planning-workflow` の Phase 1 を実行するとき

使わないケース:

- ユーザーが明示的に事前調査を不要としている場合
- 単純な文章整形だけの場合
- 法務・会計・投資判断の最終結論を出す目的の場合

## 入力として受け取る情報

親エージェントは、サブエージェントへ渡す前に以下を整理する。

- テーマ
- 背景
- 読み手・決裁者
- 期待する意思決定
- 制約条件
- 既知の候補案
- Markdown文書やIssueなどの前提資料

不足が大きく、調査観点が変わる場合は、先に最大3問だけユーザーへ確認する。

## 調査予算と網羅調査確認ゲート

事前調査では、最初からAPI・SaaS・データベンダーを網羅的に引きすぎない。特に提案が「個人利用/検証用」「v1」「PoC」「対象範囲限定」の場合は、網羅調査ではなく、意思決定に必要な反証チェックに絞る。

### デフォルト調査モード: 軽量調査

明示がなければ、以下を上限にする。

- Web検索/抽出の対象は合計5〜8URL程度まで。
- 外部API候補は代表3件程度まで。
- 法人向けデータベンダー、エンタープライズ契約、詳細料金表は深掘りしない。
- インフラ料金や周辺サービス料金は、意思決定に直結しない限り省略する。
- サブエージェント出力は1000〜1500字程度を目安にする。
- 目的は「第一候補方針を採ってよいかの反証チェック」であり、全候補の網羅比較ではない。

### 網羅調査が必要そうな場合

以下に該当する場合は、調査を広げる前にユーザーへ確認する。

- 商用公開、法人導入、外部顧客向け提案である。
- API/データライセンス/法務規約が採否の主要論点である。
- 月額費用、契約、再配信、キャッシュ、スクレイピング可否を正式判断する必要がある。
- ユーザーが「網羅的に」「全候補を」「ベンダー比較を」などと依頼している。
- 調査範囲を広げると、サブエージェントやWeb抽出のtoken/時間コストが大きく増える。

確認質問例（最大3問）:

1. 今回は軽量調査で十分ですか、それともAPI/ベンダー候補を網羅的に比較しますか。
2. 商用公開や外部顧客向け利用を前提に、利用規約・再配信・キャッシュ可否まで確認しますか。
3. 調査対象API/サービス数の上限はありますか（例: 代表3件、最大8件、網羅）。

### API/サービス調査の絞り込み手順

1. まず親エージェントが、提案の第一候補方針と比較対象を決める。
2. 第一候補を否定し得る代表的なAPI/サービスだけ調べる。
3. 料金・規約・機能は、意思決定に必要な粒度だけ確認する。
4. 調査対象を増やす必要が出たら、ユーザー確認ゲートに戻る。
5. 調査結果には「軽量調査」「網羅調査ではない」など、調査範囲を明記する。

## 調査観点

### 1. コスト

- 初期実装コスト
- 運用コスト
- 外部サービス/API利用料
- 保守・監視・人手対応コスト
- 将来の拡張時に増えるコスト

### 2. リスク

- 技術リスク
- セキュリティ/プライバシーリスク
- 法務/規約/ライセンスリスク
- 運用リスク
- ユーザー体験上のリスク
- 組織・意思決定上のリスク

### 3. 実現可能性

- v1で実現できる範囲
- PoCで検証すべき範囲
- 前提技術・API・データの入手性
- スケジュール上の不確実性
- 既存システムとの接続難易度

### 4. 既存サービス・代用手段

- 類似SaaS
- 既存OSS
- 外部API
- 手動運用
- 既存社内運用の改善
- 内製しない選択肢

## 親エージェントの集約ルール

サブエージェントの結果をそのまま貼らない。親エージェントは次の形に要約する。

サブエージェント調査がタイムアウトまたは部分失敗した場合も、すぐに停止しない。親エージェントが直接 `web_search` / `web_extract` などで主要論点を補完し、以下を分けて確認ゲートに出す。

- サブエージェントから得られた事実
- 親エージェントが補完調査した事実
- 未確認のまま残った事項
- タイムアウト/部分失敗により信頼度を下げるべき事項

部分失敗したサブエージェントの自己申告や未完了結果を、検証済みの事実として扱わない。

```markdown
## 事前調査サマリー

### 結論

### コスト見立て

### 主要リスク

### 実現可能性

### 既存サービス・代用手段

### 推奨方針

### ユーザーに確認したい判断事項
```

## agent-memory 記録ルール

`proposal-planning-workflow` の一部として事前調査を行う場合、親エージェントは作業自体を `agent-memory` コマンドで必ず記録する。サブエージェントに調査を任せた場合も、最終的な記録責任は親エージェントが持つ。

```bash
rtk agent-memory write --content "proposal research: <topic> 調査開始。観点=cost/risk/feasibility/alternatives / input=<概要>"
rtk agent-memory write --content "proposal research: <topic> 調査完了。結論=<要約> / sources=<主要URL> / 要確認=<項目> / next=<確認ゲートまたは追加調査>"
rtk agent-memory scratchpad add --text "proposal research <topic>: <未確認事項または追加調査TODO>"
```

記録する内容:

- 調査開始時: テーマ、前提資料、調査観点。
- 調査完了時: 結論、主要な根拠URL、未確認事項、推奨方針。
- ブロック時: 取得できなかった情報、代替調査、再開条件。

注意:

- 作業ログ・一時的な調査結果は daily log に残す。Hermes persistent memory には保存しない。
- `agent-memory write --target long_term` は、今後も再利用する安定した調査方針や運用決定ができた場合だけ使う。

## 品質基準

- 事実、仮定、推論、要確認を混ぜない。
- 外部情報にはURLを付ける。
- コストは断定せず、初期/運用/外部サービス/保守に分ける。
- 「既存サービスがあるなら内製不要」と短絡しない。内製理由、差別化、制約を比較する。
- わからないことは「不明」「要確認」と書く。

## Common Pitfalls

1. 事前調査なのに提案書本文を書き始める。
2. 既存サービス調査を省略し、内製前提で進める。
3. 無料APIを「無料だから低リスク」と扱う。
4. サブエージェントの出力を検証せず事実として断定する。
5. サブエージェントがタイムアウトしたのに、補完調査や未確認事項の明示なしで確認ゲートへ進む。
6. 軽量調査で足りる場面なのに、API/データベンダー/料金/規約を網羅的に調べてtokenと時間を使いすぎる。
7. 網羅調査に切り替える必要があるのに、ユーザー確認なしで調査範囲を広げる。
8. ユーザーが判断すべき事項を曖昧にしたまま次工程へ進む。

## Verification Checklist

- [ ] 軽量調査か網羅調査かを判断し、調査範囲を明記した。
- [ ] 網羅調査が必要な場合は、調査を広げる前にユーザー確認を取った。
- [ ] API/サービス候補を必要以上に増やさず、代表候補に絞った。
- [ ] コスト、リスク、実現可能性、既存サービス/代用手段を調査した。
- [ ] 事実、仮定、推論、要確認を分けた。
- [ ] 外部情報にURLを付けた、または未調査/不明と明記した。
- [ ] 推奨方針を出した。
- [ ] ユーザーに確認すべき判断事項を最大5個に絞った。
- [ ] 調査開始、調査完了、ブロック/未確認事項を `agent-memory write` または `agent-memory scratchpad add` で記録した。
