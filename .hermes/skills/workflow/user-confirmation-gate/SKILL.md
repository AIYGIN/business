# ユーザー確認ゲート

## 目的

調査結果や作業方針を提示したうえで、次工程へ進む前にユーザーの明示的な確認を取る。

特に、提案書作成、PR作成、外部公開、コストがかかる作業、スコープ変更、不可逆に近い変更の前で使う。

## 使うタイミング

- 事前調査後に、提案書作成へ進むか確認するとき
- PR作成、push、外部投稿など外部に残る操作の前
- 複数案のどれを採用するかユーザー判断が必要なとき
- 要件が曖昧で、進め方により成果物が大きく変わるとき
- `proposal-planning-workflow` の Phase 2 を実行するとき

使わないケース:

- ユーザーが既に明確に実行指示していて、追加判断が不要な通常作業
- 読み取り・調査・ローカル検証など低リスクな作業
- すぐ戻せる軽微な編集

## 確認前に整理する情報

- 現時点で理解した依頼内容
- 事実
- 仮定/推論
- 要確認
- 選択肢
- 推奨方針
- 進めた場合の成果物
- 進めない場合/別案の影響
- ユーザーに決めてほしいこと

## 標準提示フォーマット

```markdown
結論:
<推奨方針を1〜3文で書く>

根拠:
- 事実:
- 仮定/推論:
- 要確認:

選択肢:
| 案 | 内容 | メリット | デメリット/リスク | 推奨度 |
|---|---|---|---|---|

次に作る成果物:
- <成果物1>
- <成果物2>

確認したいこと:
1. <確認質問1>
2. <確認質問2>
3. <確認質問3>

この方針で進めてよいですか。
```

確認質問は最大3つにする。質問が多い場合は「今決める必要があるもの」だけに絞る。

## 確認結果の扱い

### ユーザーが承認した場合

- 承認された前提を短く復唱する。
- 次工程に進む。
- 提案書やPRなど外部成果物では、承認済み前提と未確認事項を分けて書く。

### ユーザーが修正した場合

- 修正点を前提に反映する。
- 必要なら再度、最大3問だけ確認する。
- 前提が大きく変わった場合は、事前調査に戻る。

### ユーザーが保留した場合

- 作業を止める。
- 未決事項と再開条件を明示する。

## 判断基準

確認ゲートを置くべきか迷ったら、以下のいずれかに当てはまる場合は確認する。

- コストが発生する、または将来コストを約束する
- PR、Issue、外部投稿など他者に見える成果物ができる
- 方針の違いで成果物が大きく変わる
- ユーザーの意図を推測で補っている
- 法務、セキュリティ、プライバシー、事業判断に影響する

## Common Pitfalls

1. 確認ゲートと言いながら、実質的に一択だけを押し付ける。
2. 調査結果を長く貼りすぎて、ユーザーが何を決めればよいか分からない。
3. 確認質問が4つ以上になり、判断負荷を上げる。
4. ユーザー承認前にPR作成や文書確定まで進める。
5. 仮定を事実として提示する。

## Verification Checklist

- [ ] 結論、根拠、選択肢、次成果物、確認事項を提示した。
- [ ] 事実、仮定/推論、要確認を分けた。
- [ ] 確認質問を最大3つに絞った。
- [ ] ユーザー承認前に不可逆/外部向け作業へ進んでいない。
- [ ] 承認または修正内容を次工程へ反映した。
